How to Build an API From Scratch

Created by Julian Brown
Last updated: Mar 03, 2022
23 min read17 people viewed

This is intended as a short course in how to go about building real world APIs from requirements, using DWP standards and best practice. This guide does not explain how to write code - it is assumed that you already know the basics of Javascript (or Java) and how to use the tools to make it work. Instead this is about the approaches and practices you need to use to get the job done the right way

Scenario

(Expanding the DWP engineering recruitment code test) - a data store for workers (not users as per test) and a mechanism for looking up workers according to location

There is a repository in gitlab to support this guide, providing the OAS contracts and a ready to go CI pipeline. Just take a fork of node-starter-project

Requirements

  • You need a working engineering computer

  • Node and NPM installed from self-service. The latest version is preferable

  • A suitable code editor - Atom, Visual Studio Code or Webstorm/Idea

1) API Definition

The starting point for an API should always be an API specification. As a descriptive contract definition the tool of choice here is OpenAPI Specification (OAS).

In this case the API specifications are provided but you still need to understand how to read them, fortunately the default “language” for OAS documents is yaml which is easily digestible even without specific knowledge of the OAS schema specification.

One of the easiest ways to examine an OAS document is simply to copy the contents into the online swagger editor at Swagger Editor this will translate the OAS document into a neatly formatted web page.

The three API definitions required for this project are linked on this page.

2) Why Contract Test First?

The most robust way of building our software is to start with a coded test that checks we are building the right thing. It means we should only build what we need and, more importantly, what we build is testable. With contracts this means what we build strictly adheres to the API. In order to do this there are a couple of things we can do to accelerate the initial steps

  1. Automatic generation of a server stub

  2. Automatic generation of API tests

Both of these simply need the API specifications and take just a moment to perform

Once we have the two parts we can quickly check both sides of the API contract without the need for generating extensive test scripts or populating a server stub. The point here is not to avoid the effort of developing test or code, instead we are checking that the contract is as intended.

Another significant advantage of this approach is that development of client and server code can take place in parallel without the need for constant communication from both sides. A well defined API contract should provide pretty much everything you need to build either although supporting documentation will always help ease the burden and reduce risks.

3) Code Generation (codegen) for Server Stubs

One of the greats things about using an OAS contract document is that we can automatically generate a server stub from the document. The online swagger tools include an option to generate some (very opinionated) code in the language of our choice that implements the contract as a stub. It will only return hard coded sample responses (as defined in the contract document) but it is a great starting point.

The generated code should not be used in production, it is just there for testing and verification

  • Open the swagger editor

  • Open up the workers.yaml OAS contract and copy the file contents

  • Copy the file contents into the swagger editor’s left panel, replacing anything that may already be there

  • Click “Generate Server” from the menus and select “nodes-server” from the pop-up list

  • Extract the downloaded zip-file into a safe working folder

You should now have a folder named “nodejs-server-server-generated” (yes it does say server twice). In the folder is a complete nodejs project you can build and run. You will need to perform a “npm install” to download the necessary dependencies, once finished use “npm start” to actually run the stub server.

  • In a terminal session navigate to the folder you’ve placed the code in

  • use ‘npm install’ to populate the dependencies (this may take several minutes)

  • use ‘npm start’ to start the service

  • in the terminal you should have a message telling you the address the service is running on

  • there should also be an additional address you can query, ending in /docs - this is a live version of the API contract that the code implements. If you navigate to this address from your browser you can send trial requests to the stub server and observe the results

  • when finished you can stop the server by typing ctrl-c (or control-c on a MacBook)

It’s a good idea to get this code into a separate git repository and onto gitlab. You don’t want to include it inside your api service git repo. We can expand on this later and make the stub better but for now we just need to be able to stand the server up, running locally.

4) More Code Generation for Postman Tests

Another of the great tools at our disposal is Postman. It allows us to send requests to a test server and verify the results we receive comply with the requirements. Once again we can use the API contract to rapidly create a set of tests. Between the generated server stub and the postman tests we should have a minimal working solution (minimal in that it has tests and they pass).

The automatically generated tests are simply a framework - they do not test behaviour, simply the existence of an API endpoint along with very simple checking of the API response.

There are other tools for testing, notably 'pact' which takes a more rigorous approach to encapsulating the API contract in tests. This exists alongside the direct behavioural tests rather than replacing them.

If you haven’t installed Postman, use the self-service tool to acquire a local instance. If you’ve never used Postman before go to http://postman.com and have a read of the documentation. Postman is a clever testing and exploration tool for APIs that allows us generate to ad-hoc queries for exploration of an API but it also allows us to create test collections that we can automate to verify an API that we are building

  • Open postman from the launcher

  • Don’t sign-in, just skip that step. We don’t use the online cloud version of postman. The local version is just fine

  • Click the import button and select OpenAPI from the options

  • Browse to the workers.yaml contract and click open

  • You can modify the defaults but that shouldn’t be necessary right now, just click “import”

  • Select collections from the left hand palette column

  • Expand “worker store”

You should see a set of endpoints and methods, each one relating to one of the definitions in the OAS contract

If you click on “worker store” and select variables it should be pre-populated with a set of relevant values, all taken from the contract

  • make sure the stub service is running (by default it should start on port 8080 as per the contract)

  • in postman click on the “GET /workers" test. The right panel should show a new tab for the default /workers path using the GET method

  • click the “send” button on the GET /workers tab

  • postman will send a new request to the stub and collect the response back. It should be a 200 response with an empty list of workers

The ugly part of Postman is that you have to export the changes to save them in git, and keep doing so with each round of changes

  • Hover your pointer over the “worker store” collection in Postman, a star and ellipsis should appear to the right of the collection name.

  • Click the ellipsis and select export from the pop-up menu

  • Click export in the export window

  • Navigate to your API service repository, it should start from your resources folder by default as that is where it loaded the API spec from. Go up a directory level if the resources folder is the starting point and create a new folder “postman” in the root of the repository. Export the collection into the postman folder

  • Add the new collection file to git and commit it to the repo

  • Don’t forget to stop the stub service

5) Automate All The Things

Before we write any of our own code we need a way to automatically check what we write is a) working; and b) still meets the requirements. The tool to use here is a CI pipeline. We use Gitlab as our code repository and it allows us to define a CI pipeline with very little effort. The engineering team has already created reusable CI fragments that can be pieced together, all we need to do is import those pieces and make sure the overall configuration is ok. Gitlab will even check the definition without having to run the pipeline!

  • assuming you have a fork of the starter repository, push the freshly committed code to gitlab

  • navigate to the gitlab page for the repository and you should see there is a pipeline running already

  • the pipeline will likely fail at this point as you don’t actually have any code

The pipeline will run every time you push code to the repository, it checks the code in a lot of different ways to make sure it meets our standards and passes all the tests we create

6) And do it all again…

We need to have all three API contracts to the same level. You have the other contracts so repeat the process for each of them

At the end we should have three git repositories, one for each API, and another three repositories for the three stub services that you’ve created

7) Where to Start with a Microservice

In microservice architecture we aim to keep things small and isolated. A set of data is kept away from other data and logic on how to utilise the data is kept separate. For example, all a data service needs to do is store, retrieve, update and delete the data (aka CRUD).

Microservices work in composition with other services to create patterns of logic and behaviour rather than cramming all of the work into one single place.

Each service needs to consist of a few things - first and foremost is the API definition (see above); second is an implementation of the service endpoints; finally comes internal logic. We’ve already described the first two - although in the case of the endpoints all we have is a stub that meets the requirements without the internal logic. We even have a set of tests that check that the endpoints are as described. The next step is to code our own endpoint shell to meet the test requirements. Once we have that we can look at the internal logic, coding more detailed unit tests for this without needing to know too much about how the service behaves from the outside.

Typically a microservice will also need to interact with other (upstream) services to acquire information, or a data store to store information. Where the information is upstream the service will need to implement a service client - again this should come from an API document. We can even use swagger tools or postman to create a basic client that allows us to test the upstream service. For a data store we need to interact with a local data api to store and retrieve information.

There are other ways to interact, such as message queues and events but for the purposes of this course all of the interaction will be using a REST approach to services.

Start with looking at the postman tests

8) Expanding Postman

The set of tests we already have are very minimal, they just prove that an endpoint exists and communicates with the right format of data. If we want to check behaviour we need to expand the tests. We can either provide more detailed tests or use another test framework (e.g. “REST assured”) to compose more detail and describe behaviour.

If we stick with postman we need to create test scripts that verify specific test scenarios - so we need to define what those scenarios (or “user journeys”) are. We aren’t looking for a huge number of tests at this level, just enough to describe how we want the service to behave.

Postman tests can be duplicated to expand the collection and to keep different scenarios separate. We also need to make sure we only add one test at a time and make sure it is implemented as a solution before starting on the next.

If you examine one of the postman endpoint tests you will find it has a further layer of detail, describing the different responses the endpoint might give. These need to be turned into tests rather than descriptions. The key is in describing behaviour, not function. We could compose a test that checks that an error response is returned but it wouldn’t describe why the error response should be returned, instead we need to provide a scenario that details the pre-existing condition, the interaction with the service and what we expect (aka Given, When, Then).

The test needs to be fixed, as in, given the same starting point and request we should always see the same response. This means we have to control that starting point, we must not rely on other tests to set the scenario up. The test must be completely self contained. When tests are created this way we can run any test, in any order and it will still work.

How we control the starting point depends on the type of tests we are executing. Right now we have just a stub so we have total control of each endpoint so we can tell the stub to return a specific response when supplied a similarly specific request. This also means that each and every test scenario for each endpoint must have a unique set of request data, failure to keep this separation intact results in broken tests later.

  • In postman expand the Worker Store collection and navigate through the tree to “GET /workers/:workerId”

  • There is a single “test” with three examples - we need a test per example so duplicate the existing test twice - right click the test and select duplicate from the menu

  • Rename each of the three tests to represent the expected behaviour, following the examples

    • GET /workers/:workerId - success finding worker

    • GET /workers/:workerId - invalid Id supplied

    • GET /workers/:workerId - worker not found

  • The three tests will still have the same examples but we can ignore that

  • Select the first of the tests and then select the “params” tab. The parameters are split into two sections, query and path. This endpoint doesn’t have any query parameters but it does have a path parameter - workerId

  • By default the workerId key value is pre-populated but it is the same value in all the tests so make sure it is changed to a unique numeric value here

  • Select the tests tab - this is where the coding begins

  • We want to check that we see a 200 response, conveniently there is a pre-written fragment of code we can copy in, to the right of the pane is a list of snippets. If you scroll down through the list you will see “Status code: code is 200”. Click on this and three lines of code will be placed in the test pane:

    1pm.test("Status code is 200", function () { 2 pm.response.to.have.status(200); 3});

    This test comes in two parts, first a description of the intent of the test as plain text “Status code is 200”; second is an assertion “pm.response.to.have.status(200)”. The assertion describes how to actually check the response is as expected

  • We can repeat this process for each of the other two tests but we need to modify the test code to match the intent. The invalidId test needs a unique workerId that is not a number and should have a response of 400, there is no exact snippet for this but we can reuse the “code is 200” test by just changing 200 to 400. The worker not found test just needs a unique numeric number, the response should have a code of 404

  • If you run the tests now the two new tests will pass one fails - this is what we would expect from an unmodified stub. The invalid id test is already caught by the schema validator baked into the stub code. We still need to modify the stub to match the expected behaviour

Note we are only testing the response status codes here for simplicity, the actual tests needed for a real service are more complex and should include at the very least response schema validation and simple assertions for content of the response, like the workerId matching the one given in the request.

Do not forget to export the postman collection to your git repository and commit it!

8b) Expanding the Stub

The default code from swagger-codegen is pretty basic and only provides a simple response, we need to expand this. It is only small changes but right now we don’t need more.

Open up the nodejs-server-server-generated folder for the worker store service and expand the folders to make things a little clearer. If you haven’t performed the npm install do that step to make sure all the service dependencies are in place

  • In the stub service folder open up ‘DefaultService.js’ from the service folder

  • after the use strict line add an import to include the provided utilities code:

    1var utils = require('../utils/writer.js');

    This enables use of the expanded response we need for our tests

  • Go to the end of the file and find the workersWorkerIdGET function - this is where the stub service resolves requests to the workers/:workedId path

  • If you examine the code you’ll see that the code has an example response, we want to modify things so that the stub behaves in a more complex manner and can return something other than a 200 status code. Right now it just returns the example if any workerId is supplied:

    1if (Object.keys(examples).length > 0) { 2 resolve(examples[Object.keys(examples)[0]]); 3} else { 4 resolve(); 5}

  • We want to place a wrapper around this to handle the specific scenarios we are providing. Start by placing an if-else block ahead of it so that the existing code is included in the else scenario. The if clause needs to match the workerId to the one given in our not found test scenario

    1if (workerId === 80521650) {

    The exact id is up to you - but it must match exactly

  • The body of the if statement needs to generate a full response as described in the OAS contract, in this case it is a status code of 404 and an empty response body

    1resolve(new utils.respondWithCode(404, ));

  • If you restart the stub service and run the postman tests, all three tests should pass but our tests aren’t very nice in their current state and not really up to our coding standards so we can refactor the code into something more readable and explicit

    1switch (workerId) { 2 case 80521649: 3 resolve(examples[Object.keys(examples)[0]]); 4 break; 5 case 80521650: 6 resolve(new utils.respondWithCode(404, )); 7 break; 8 default: 9 if (Object.keys(examples).length > 0) { 10 resolve(examples[Object.keys(examples)[0]]); 11 } else { 12 resolve(); 13 } 14}

  • Restart the server and re-run the tests to make sure everything still passes as expected

We’ve covered off some expanded tests but we need to repeat this for all of the scenarios in our suite of postman tests across all three services. The steps are broadly the same and this is just a matter of making sure that we testing all of the expected current behaviours

It is important to note we are modifying the behaviour of generated code - if we re-run the code generator and overwrite our stub server, we would lose all of these changes. There are other ways to achieve much the same behaviours in a safer manner. It is worth thinking about this and planning out how to make the changes persistent. Once again we are using a simple and convenient approach for the purpose of this exercise

9) Test First Code

The method of choice for building code is TDD (test driven development). This is repeating the pattern of creating a test, implementing the code and making sure it works before moving to the next objective. Where the postman tests are few in number, TDD tests (unit tests) are many but smaller and more focussed. Each test needs to add just one piece of detail rather than a whole user journey

We can start any part of the relevant logic that aligns to the bigger postman test we are working against (note: not all requirements in the real world will need a big behavioural test behind them)

The aim of the TDD approach is to create a robust suite of tests that we can run automatically, reliably and in any order - all to describe how the logic of our service should behave. The important consideration is that we should be able to re-implement our code without having to rewrite the tests. This is “black box” testing because it doesn’t know how the code works internally. If we have to rewrite our tests to suit our code we have a problem, the tests have become fragile!

We should also be running the tests with each bit of code added. We can run the unit test suite locally, and the postman tests but ideally we should be using the CI pipeline and that means committing our code frequently to the git repository to trigger the process. Keep in mind that the pipeline typically takes several minutes to run so while committing code to git should happen with each step, pushing the code to the remote repository doesn’t have to be quite so frequent

In practical terms you need to start with a clean project, do not be tempted to re-use the generated server code. It might be convenient but we need our code to be fully testable and that means writing tests and only writing code for satisfying those tests. You already have a postman collection though that you can use, import that into your project repository.

9a) Keeping Postman Happy

Since we have tests already we can start there - we aren’t going to be able to satisfy all of the tests in one go though. Creating a service that provides a single endpoint allows you to produce something useful - the GET /workers endpoint is probably the easiest since it just returns a list and we can hard-code that list for the purpose of delivery and passing the postman test while the service is extended to include other functions.

Once you have an endpoint for GET /workers the next logical step is the create endpoint at POST /workers, this starts to push the implementation into achieving useful functionality as a data store and extend past simply returning a fixed list of workers. Implement the endpoint first with a fixed stub response so that it passes the relevant postman test. Now we can actually build some real code.

In terms of “units” we need a data transfer object (the worker data structure) that can be readily created from the request json using a builder. The contents of the object should be considered immutable since there is no need to modify it (even the update function is just overwriting rather than modifying). We don’t need any set accessors to the data, just gets. So out of this we have two units that need coding - the data object and a builder for it.

We need a mechanism for pushing the data to a persistence system (aka database) and a matching mechanism for retrieving the data. It is tempting to put all this in the data object but the data object doesn’t need to know anything about the database and vice-versa so we need a data handler to keep the data object and database abstract from each other. There are ready made dependencies for this that generalise and completely abstract away the database implementation or more native drivers that only work with a single database type. There are arguments for and against both options - if we need performance the native drivers are generally the better idea but need more coding to achieve. If we want to keep coding simple and standard then the general data handling approach helps to improve how long it takes to implement. The important bit is that the choice should be transparent - you just need 6 operations from the data handler along the lines of the CRUD paradigm. The retrieve option has three versions - get all, get one, and get all filtered by “home”. This effectively is your interface to the data handler. Compose the data handler as a wrapper around the actual implementation and you can then pick and choose how you implement this process without impacting the other objects in the design.

Initially we just need the create and basic retrieve options - the rest can wait.

Remember in order to create and test the objects in our code we must limit interaction so it is crucial to mock the presence of objects outside of the unit under test.

Once you have the POST /workers endpoint implemented from request through to data store it is simple enough to return to the GET /workers endpoint and implement the missing parts of the process here but this presents another issue. In order to test the GET operation we need data but the database should be empty before we start testing the POST operation and you don’t know which test goes first (it shouldn’t matter). The answer is to make sure we have an external mock of the data store while we perform tests from postman in much the same way we had a stub service to begin with - the mock store needs to respond with the right data for each test. Ready made dependencies exist for this purpose like mongo-mock for mongodb. You still need to make sure that the database is in the correct state for each test by setting the entry conditions before the test and cleaning up after the test, this can take some extra code to perform those tasks, particularly in postman where it doesn’t have native access to the data store.

Once you have the two basic operations in place it is just a matter of iterating through the different endpoints and outcomes to make sure everything is implemented as needed.

The findByHome option is peculiar in that it doesn’t fit the standard pattern but still has to be accommodated in the design. There are two fundamental approaches to tackling the issue of filtering objects by a specific field. We can either perform the task internally, or off-load the task on the data store. The important consideration is that “knowledge” of the data object is correctly contained. In the simplest form it would be the data object has the logic to decide if it meets the requirement or not but this forces the capability to be internal while the database is likely to be significantly more performant at selecting matches.

10) Rinse and Repeat

As with all agile development practices this is an iterative process. Once we have written the code that satisfies a unit test, we write the next unit test. On and on until we’ve satisfied the bigger behavioural test. Then we write the next behavioural test, write the unit test, write the code, etc.

Typically each behaviour will have its own work ticket in Jira so the objective is to complete one ticket at a time.

We should also be practicing proper CI - that means doing work in small enough chunks that we can merge it back to the main code branch without holding on to it for long periods (more than a day). Deciding whether a piece of code is fit to merge back in is down to two main factors. 1) does it completely pass the CI pipeline; and 2) has it been reviewed by another engineer. Traditionally we performed big code reviews for each merge but if we are doing CI properly we should be working in pairs and we should be reviewing code as we go. The only time a code review is needed should be when you get stuck and need assistance; or just need some guidance. If you are following best practice your code should be ready without too much input - that means using TDD, coding with SOLID principles and checking the fitness of code using the CI pipeline!

11) Stubbing Upstream

One of the key points of unit testing is making sure you only test the code you are working on, not the whole thing. This means, for a given class, you are mocking the interactions with other classes. If you need to run the whole service to test it then you aren’t doing unit testing. When it comes to the behavioural tests we stop talking about classes and start talking about services. You don’t want to run lots of services in parallel in order to perform a single test, at least not the real ones - so we mock those services again. We can either use the codegen server stubs with the minimum data added to satisfy the tests or we can use a more general tool, WireMock that covers the behaviour of all of those services in one place. All you need to do is define the upstream request and what the response is, WireMock will do the rest.

In an existing live system you can use WireMock to record the interaction between a mock client and the service; you can then use that recorded response as a playback when testing. Both approaches are important - you need the responses to your tests to be exactly what you need but you also need to prove that the responses you use are live-like and simply relying on documentation can lead to interesting (poor) results. Something as simple as getting the case wrong on a field name can break the system!

So for those behaviour tests you can get away with a tiny number of active services, all run locally that allow you to run the tests and have verified results

12) Containerisation

Another major tool in our armoury is containers - these allow us to guarantee that the same code is run in the same way, in the same environment every time. Not just in your local development environment.

Historically we’ve used docker desktop to provide container services but we need to break free from that tool in the not too distant future. For now though we can provide a docker build file to define how to build a container and a docker-compose file to describe how to assemble services in combination for testing (ie. our service and WireMock)

13) Logging

There are additional, non-functional requirements we have to satisfy. One of these is logging - we need to make sure we have some record of what our service is doing internally. The main purpose is to give us intelligence on when something has gone wrong as can’t observe code in the live environment

There are few things we need to do here:

  1. Make sure we have a way to correlate each logging statement with a request - better still make sure we can correlate all the requests in a chain of behaviour

  2. Do not record personal information. This makes logging tricky as it really limits what can be placed in logs as pretty much all of what we handle is personal information

  3. Provide a usable trace of how a request was handled and the response outcome, what other services were called and the outcome of those requests

  4. Ensure that each log entry has value. Ideally each log contains unique detail - logs of “started ‘this’” are just useless. When you search the logs it simply becomes noise. Fortunately most logging systems allow us to include details such as timestamps and correlation values automatically and implicitly so we don’t have to manually include them in each statement

  5. Keep all log output in a structured format such as JSON to make aggregating and searching for information much, much easier. We also need to make sure we use the right logging level for the log we are providing

14) Metrics

The second big non-functional requirement we need to include is metrics. The important point here is that logging is not metrics. The results are captured independently and are much, much more focussed. We also need to capture details about the platform, not just the progress of a single request. Metrics are formed around events rather than logs, along with monitoring of how a service is performing and behaving.

An event might be the number of requests received, how many result in success and how many are failures how long it takes a request to be processed, how long it takes an upstream process to perform its' task, the proportion of good and bad responses, etc. There is clearly overlap here on the logging but we shouldn’t be using the logging to provide the metrics.

Monitoring of the platform also needs to take place, memory and CPU utilisation are typically recorded along with storage volume. All of this is to the aim of making sure our service is healthy. If we see overload we can scale the service up, providing more instances. Likewise if the services are idle we can scale down to save on the cost of running services.
